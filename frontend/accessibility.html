Accessibility Statement

SecTech is committed to providing a website that is accessible to the widest possible audience, regardless of circumstance and ability.

Commitment to Accessibility

We aim to adhere as closely as possible to the Web Content Accessibility Guidelines (WCAG 2.2), published by the World Wide Web Consortium (W3C). These guidelines explain how to make Web content more accessible for people with disabilities. We believe that accessibility is a fundamental right and a responsibility of website providers.

Measures Supported

SecTech takes the following measures to ensure accessibility of our website:

  • Testing accessibility using the external tool barrieretest.at
  • Manual testing using our WCAG 2.2 AA checklist

Conformance Status

The Web Content Accessibility Guidelines (WCAG) defines requirements for designers and developers to improve accessibility for people with disabilities. It defines three levels of conformance: Level A, Level AA, and Level AAA. We aim to make our website conformant with the WCAG 2.2 AA level.

We regularly test accessibility of our page with various tools such as barrieretest.at. As of 04.01.2026, all of our pages achieve a perfect accessibility score of 100/100.

Feedback

We welcome your feedback on the accessibility of the SecTech website. Please let us know if you encounter accessibility barriers on our site.

WCAG 2.2 AA Checklist

Last modification: 04.01.2026

1. Perceivable

1.1 Text Alternatives

  • Do all informative images have an alt attribute?
  • Do all informative videos have captions?
  • Do all informative audio files have transcripts?
  • Are CAPTCHAs accessible?

1.2 Time-based Media

  • Are time-based media elements (e.g., videos, audio) accessible?
  • Are subtitles complete and synchronous?
  • Are captions complete and synchronous?

1.3 Adaptable

  • Are headlines structured hierarchically?
  • Are lists and tables correctly labeled?
  • Are form fields and lables correctly associated?
  • Is the reading order in the DOM logical and meaningful?

1.4 Distinguishable

  • Is text contrast at least 4.5:1 (large text >= 3:1)?
  • Is information not conveyed by color alone?
  • Can content be zoomed to 200% without loss of functionality?
  • Can moving, blinking, or auto-updating content be paused or stopped?

2. Operable

2.1 Keyboard Accessibility

  • Is all functionality accessible via keyboard?
  • Is there enough time for users to read and use content?
  • Are there no keyboard traps?
  • Is the focus order logical and predictable?
  • Can modal dialogs be closed with ESC?

2.2 Seizures & Physical Reactions

  • Is there any flashing, potentially seizure-triggering content?
  • Is there any content that could cause physical reactions?

2.3 Navigation & Orientation

  • Do pages have descriptive and unique titles?
  • Are links clearly identifiable?
  • Is navigation consistent across pages?
  • Are navigation elements (e.g., menus, buttons) clearly labeled?

2.4 Focus

  • Keyboard focus is not hidden by sticky headers, cookie banners, etc.
  • Focus indicator is clearly visible
  • Interactive elements are at least 24x24 CSS pixels, or adequately spaced
  • No accidental activation from touch or pointer gestures

3. Understandable

3.1 Readable

  • Is the primary language correctly defined in the HTML?
  • Are language changes properly marked?
  • Is the text clear, simple, and understandable?

3.2 Predictable

  • No unexpected context changes (e.g. auto-submit)
  • Consistent behaviour of navigation and controls

3.3 Input Assistance

  • Are required fields clearly indicated?
  • Are errors described in text, not only by color?
  • Are clear suggestions provided to fix input errors?
  • Are error messages programmatically associated with fields?
  • Is user input not lost without warning?

4. Robust

4.1 Compatibility with Assistive Technologies

  • Does the site work with screen readers (e.g. NVDA)?
  • Do all interactive elements have a correct role, an accessible name, and a correct state?
  • Is ARIA not misused or applied unnecessarily?
  • Do custom controls behave like native HTML elements?
